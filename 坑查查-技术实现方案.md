# 技术实现方案：「坑查查」生活防坑与常识科普平台 V2.0

> 项目名：**坑查查（KENG CHACHA）**
> 文档版本：V2.0　|　配套：《坑查查-产品需求文档PRD.md》
> 本文明确：**技术栈选型、系统架构、微服务划分、数据库设计（含 DDL）、AI 能力落地、数据采集流水线、2 小时定时更新与一致性、安全合规、部署运维**。

---

## 1. 总体技术目标

1. **PC + H5 双端**，统一后端，内容强一致；为微信小程序预留扩展。
2. **每 2 小时滚动更新**，高可用、强一致、无感知。
3. **AI 全链路赋能**：内容生产、对话问答、风险检测、自适应测评、个性化推荐、语音/OCR/文生图。
4. **低门槛交互**：语音输入/搜索/播报、拍照识别、对话查询。
5. **合规优先**：境内合规大模型、内容安全审核、隐私加密、算法备案。
6. **可演进**：微服务化、可水平扩展、AI 能力可插拔、可降级兜底。

### 2.6 开发阶段轻量化替代方案（最小资源占用，推荐）

> 上表为**生产级**目标架构。开发/自测阶段一台笔记本难以同时跑起 Nacos + 网关 + ES + Milvus + MongoDB + RocketMQ + ClickHouse + K8s（轻松吃满 16G+ 内存）。
> **开发期原则：用"一个进程 + 一个 PostgreSQL + 一个 Redis + 云 AI API"覆盖几乎全部能力，把 10+ 中间件压到 2~3 个。** 接口与代码结构保持不变，仅替换底层实现（Spring Profile + 抽象接口隔离），后续可平滑切回生产组件。

| 用途 | 生产方案 | 开发期轻量替代 | 省资源点 |
|------|----------|----------------|----------|
| 整体架构 | 微服务 + Nacos + Gateway + Sentinel + Seata | **Spring Modulith 模块化单体**（单进程、模块边界清晰、零注册中心/网关） | 省多进程 + 注册中心内存 |
| 关系库 | MySQL 8 集群 | **PostgreSQL 16 单实例**（起步亦可 H2/SQLite） | 一库同时承担 关系 + 向量 + JSON + 全文 |
| 向量库 | Milvus | **pgvector**（PostgreSQL 扩展）或 Chroma（嵌入式） | 免独立向量集群 |
| 文档/非结构化 | MongoDB | **PostgreSQL JSONB 字段** | 去掉 MongoDB |
| 全文检索 | Elasticsearch | **Meilisearch**（单二进制、中文友好、几十 MB）或 PG 全文（`zhparser`） | ES 常占 1G+ → 数十 MB |
| 缓存 / 分布式锁 | Redis Cluster | **Redis 单节点**（内存 ~10MB） | 已极轻，单点足够 |
| 消息队列 | RocketMQ / Kafka | **Redis Stream** 或 **应用内事件**（Spring `@EventListener`）；需独立 MQ 时用 **NATS**（单二进制） | 免 ZK / Broker 重组件 |
| 行为分析 | ClickHouse | **DuckDB**（嵌入式 OLAP）或先落 PG；dev 可暂不接 | 省 OLAP 集群 |
| 对象存储 | 阿里云 OSS | **本地文件系统** 或 MinIO 单容器 | 免云依赖 |
| 大模型 LLM | 云 API + 私有化 Qwen（vLLM/GPU） | **纯云 API**（零本地占用）或 **Ollama + Qwen2.5-0.5B/1.5B 量化**（纯 CPU 可跑） | dev 无需 GPU |
| Embedding | 自部署 BGE-large | **bge-small-zh**（百 MB 级，CPU 友好）或向量化云 API | 显存/内存大幅下降 |
| ASR / TTS / OCR | 云语音 + 自部署 | **纯云 API 按量调用** | 零本地模型 |
| 编排部署 | Kubernetes | **Docker Compose** + 本地 `./gradlew bootRun` | 免 K8s 控制面 |
| 监控观测 | SkyWalking + Prometheus + ELK | **Spring Boot Actuator + 控制台日志**（按需加单实例 Grafana） | 省一整套观测栈 |

**开发期"最小启动清单"（内存预算 < 1.5GB）**

```
仅需 3 样：
 1) PostgreSQL 16（装 pgvector 扩展）  ← 关系 + 向量 + JSONB + 全文 四合一
 2) Redis 单节点                       ← 缓存 + 锁 + Stream 当轻量队列
 3) 后端 Spring Modulith 单体 (bootRun) ← 业务全模块同进程
 +  云 AI API Key（LLM / ASR / TTS / OCR 按量，不占本地资源）
 +  对象存储用本地 ./uploads 目录
```
> 一份 `docker-compose.dev.yml` 起 PostgreSQL + Redis 两个容器即可；前端 `vite dev` 直连后端。
> AI 离线开发可选 Ollama 跑 Qwen2.5-1.5B（约 1~2G 内存）做基本问答联调。

**保证可平滑切回生产的工程约束：**
1. **依赖抽象**：存储/检索/MQ/对象存储/AI 全部走接口（如 `SearchPort`、`VectorStore`、`EventBus`、`BlobStore`、`LlmClient`），dev 与 prod 各提供实现，靠 **Spring Profile（`dev` / `prod`）** 切换，不改业务代码。
2. **SQL 兼容**：dev 用 PostgreSQL、prod 若用 MySQL，则用 **Flyway/Liquibase** 管理双方言迁移，避免数据库特有语法散落代码（或干脆 prod 也用 PostgreSQL，统一最省心）。
3. **模块即未来的服务**：Spring Modulith 的模块边界（user / content / quiz / ugc / assistant / toolbox …）就是日后拆分微服务的切割线，单体→微服务零返工。
4. **AI 接口统一**：`ai-service` 在 dev 可与单体同机部署或仅用云 API；prod 再独立成 Python/FastAPI 服务，调用方契约不变。

> 建议：**dev 与 prod 都用 PostgreSQL**（pgvector + 全文一站式），可省去 ES/Milvus/Mongo 三套组件，到中后期数据量/检索压力上来后，再按需把"全文"迁 ES、"向量"迁 Milvus、"文档"分流，演进式升级而非一步到位。

## 5. 数据库设计

> 说明：以下为核心表结构（MySQL DDL，省略部分审计字段）。统一约定：所有表含 `id BIGINT 主键`、`created_at`、`updated_at`、`is_deleted TINYINT` 逻辑删除；敏感字段（手机号、私密内容）**字段级 AES 加密 + KMS 托管密钥**；高频读对象走 Redis 缓存。


```sql

-- 内容-标签关联(多对多)
CREATE TABLE `content_tag` (
  `content_id` BIGINT NOT NULL,
```

### 5.6 非关系型存储设计

**MongoDB（data-pipeline / assistant）集合：**
- `raw_crawl`：原始抓取内容（仅官方源原文 + 社区源线索摘要，含来源、时间、原始链接）；社区源**不存全文**，仅存提炼后的线索字段。
- `ai_intermediate`：AI 加工中间产物（抽取结果、改写草稿、打标置信度）。
- `assistant_conversation`：AI 助手会话与消息（user_id、轮次、输入模态、检索命中、回答、用户反馈）。
- `toolbox_raw`：检测原始输入暂存（合同 OCR 文本、链接快照），按隐私策略 TTL 过期删除。

**Elasticsearch 索引 `pitfall_content_idx`（检索）：**
```json
{
  "title": "text(ik_smart)",
  "body":  "text(ik_smart)",
  "trick": "text", "tip": "text",
  "stage_tags":  "keyword",   // 人生阶段
  "domain_tags": "keyword",   // 领域
  "psych_tags":  "keyword",   // 心理机制
  "channel_tags":"keyword",   // 渠道
  "hazard_level":"byte",
  "region_code": "keyword",
  "hot_score":   "integer",
  "online_at":   "date",
  "embedding":   "dense_vector(1024)"  // 语义检索(或交由 Milvus)
}
```

**Milvus 集合 `content_vectors`：** `content_id` + `embedding(1024, BGE)` + 标量过滤字段（hazard_level、stage、domain），用于 RAG 与"相似案例"。

**Redis 关键 Key 设计：**
| Key | 用途 | 类型/TTL |
|-----|------|----------|
| `feed:home:{region}:{page}` | 首页信息流缓存 | List，2h |
| `refresh:countdown` | 距下次更新倒计时(下次时间戳) | String |
| `rank:contrib:weekly` | 贡献榜 | ZSet |
| `hot:content` | 热门内容 | ZSet |
| `session:{token}` | 登录会话 | Hash，过期续期 |
| `lock:refresh:batch` | 更新任务分布式锁 | String(SETNX) |
| `limit:{api}:{uid}` | 接口限流计数 | String，滑窗 |
| `assistant:ctx:{sessionId}` | 助手多轮上下文 | List，30min |

## 6. 数据采集与内容生产流水线（核心）

```
[1] 采集 Crawler/Connector(Python, 按 data_source 配置)
      官方源: API/RSS/合规抓取  →  社区源: 仅取"线索摘要"
                 │ (写 raw_crawl, 发 MQ: topic=raw.collected)
                 ▼
[2] 清洗去噪(去广告/导航/水印/HTML)  + 去隐私(正则+NER脱敏)
                 │
                 ▼
[3] AI 抽取(LLM): 四要素 + 五维标签 + 危害等级 + 损失区间
                 │ (写 ai_intermediate)
                 ▼
[4] 去重聚类(向量相似度 + SimHash) + 可信度评分(多源交叉)
                 │  低可信/重复 → 丢弃或合并
                 ▼
[5] AI 改写为原创科普(<=200字)+ 生成口诀(规避版权/隐私)
                 │
                 ▼
[6] AI 文生图/GIF/信息图(套模板) + TTS 生成播报音频 → OSS
                 │ (media-service, 发 MQ: topic=media.gen)
                 ▼
[7] 内容安全审核(文/图) + 人工抽检(后台审核台)
                 │  通过 → status=待上架, 入 ES/Milvus 索引
                 ▼
[8] XXL-JOB 每2小时: 组批 → 灰度 → 全量上架 → 刷新缓存/倒计时
```

要点：
- 各阶段通过 **RocketMQ** 解耦，失败重试 + 死信队列；幂等以"来源指纹"去重。
- 社区源**只发现线索、AI 重写为原创**，从不入库或展示原文，规避版权与隐私风险。
- 可信度评分低于阈值不进前台；涉未决事件强制"涉嫌/疑似/请核实"措辞校验。

## 7. AI 能力落地细节

### 7.1 AI 避坑助手（RAG + Agent）
- **检索增强问答（RAG）**：用户问题 → Embedding(BGE) → Milvus/ES 混合检索 Top-K 案例与知识 → 拼装 Prompt → LLM 流式生成。
- **多模态前置**：语音 → ASR 文本；图片/截图 → OCR + 图像理解转文本；链接 → 抓取快照转文本，再进入 RAG。
- **Agent 工具调用**：助手可调用「链接验毒」「企业核验」「合同体检」等工具（Function Calling），实现"边问边检测"。
- **Prompt 约束（系统提示要点）**：
  - 只做防御性科普；拒绝输出任何"实施/规避监管"内容。
  - 不臆造，未知则说明并给官方核验渠道（96110/110/12315/12348）。
  - 回答结构固定：是否是坑 → 套路类型(关联分类) → 套路推演 → 你该怎么做 → 关联案例。
  - 输出末尾固定免责声明："AI 辅助判断，不构成法律或官方结论"。
- **降级兜底**：AI 不可用时回退关键词检索 + 高频 FAQ。

### 7.2 自适应测评（IRT）
- 题目带 `difficulty/discrim` 参数；基于**项目反应理论**实时估计用户能力 θ，答对升难、答错补弱维度；终止条件为题量上限或能力估计收敛。
- 画像与建议：由维度得分映射趣味标签库 + LLM 生成个性化"高危场景预测/可执行建议"。

### 7.3 视觉素材自动化（文生图 + 模板）
- 结构化要素 → 模板引擎（固定版式：标题/套路/损失/口诀，重点高亮）+ 文生图生成背景/插画 → 合成海报/信息长图；GIF 由关键帧脚本生成。
- 模板化保证 PC/H5 多分辨率排版一致，水印 + 防伪二维码。

### 7.4 风险检测工具
- **合同体检**：OCR → 条款切分 → LLM 逐条对照"风险条款知识库"（空白条款、违法竞业、定金不退、高额违约金等）→ 标红 + 解释 + 建议。
- **链接验毒**：域名信誉库 + 仿冒检测（相似域名/品牌）+ 页面内容 LLM 研判 → 风险等级。
- **拍照识坑**：图像理解 + 危险话术分类模型（"先付款""走私人账户""屏幕共享""征信修复"等）→ 命中套路 + 高亮。
- 所有工具输出附"非结论"声明与官方核验入口。

### 7.5 内容审核与隐私
- 文本/图片走云内容安全；人脸、证件、车牌、住址等**自动检测打码**后再存储展示；私密数据字段级加密。

### 7.6 模型工程
- 云 API 为主、私有化 Qwen（vLLM）为备，按成本/合规/时延路由；异步重任务走 Celery + MQ；统一 `ai-service` 封装，便于替换与灰度。

## 9. 2 小时定时更新与一致性（关键非功能）

1. **调度**：XXL-JOB 每 2 小时触发 `content-refresh-job`；分布式锁（Redis SETNX）保证单实例执行，失败自动转移。
2. **组批**：从"待上架池"按热度/多样性/地域均衡选取一批，写入 `content_refresh_batch`。
3. **灰度→全量**：先灰度小流量验证素材/排版，再全量切换 `status=已上架`，更新 ES、Milvus、Redis 信息流缓存。
4. **倒计时一致**：`refresh:countdown` 写入"下次更新时间戳"，PC/H5 统一拉取，前端本地仅做秒级递减，保证多端一致。
5. **强一致与无感知**：采用"双缓冲"——新批次预热完成后原子切换缓存指针，用户无卡顿；切换失败自动回滚到上一批。
6. **可观测**：每次更新记录上架数量、耗时、失败项，异常告警（更新延迟/素材缺失/审核未过）。

## 10. 安全、风控与合规

- **传输与存储**：全站 HTTPS；敏感字段 AES-256 字段级加密，密钥 KMS 托管；对象存储私有读 + 临时签名 URL。
- **鉴权**：JWT + 刷新令牌；微信网页授权；接口签名防篡改防重放。
- **风控反作弊**：注册/登录风控、设备指纹、行为验证码；UGC/点赞限频、查重、刷量识别；爬虫防护（WAF + 频控 + 蜜罐）。
- **内容合规**：AI + 人工双审；谣言/涉敏/点名攻击拦截；未决事件审慎措辞；投诉与一键下架机制。
- **隐私合规**：遵循《个人信息保护法》《数据安全法》；最小收集、明示同意、可导出可删除；检测/测评/上报私密数据默认加密、不外泄；佐证图自动打码；toolbox 原始输入设 TTL 自动清理。
- **AI 合规**：境内合规大模型、算法备案；AI 生成内容显著标识；安全护栏（拒绝助纣类请求）。
- **未成年人保护**：相关内容审慎、引导监护人、不收集非必要未成年人信息。

## 11. 性能、高可用与扩展

- **缓存策略**：信息流、热门、倒计时、画像结果多级缓存（本地 Caffeine + Redis）；热点探测防击穿/雪崩（互斥锁 + 逻辑过期）。
- **检索**：ES 负责标签过滤 + 全文，Milvus 负责语义；可并行混合召回再融合排序。
- **异步化**：素材生成、AI 重任务、审核、推送全走 MQ，削峰填谷。
- **虚拟线程（Java 21）**：网关与内容/助手等"高并发、IO 密集、调用 AI/ES/下游"的服务开启虚拟线程（`spring.threads.virtual.enabled=true`），以同步代码写法承载海量并发连接，降低线程池调参成本。
- **水平扩展**：无状态服务多副本 + HPA 自动扩缩容；MySQL 读写分离 + 分库分表（UGC/行为表按 user_id 哈希）；ClickHouse 承接海量埋点。
- **可用性**：核心链路熔断降级（Sentinel）；AI 服务故障回退关键词检索；多可用区部署。
- **成本**：AI 调用按场景路由（小模型/规则前置过滤，大模型仅处理复杂任务）+ 结果缓存（相同问题/相同链接命中缓存）。

## 14. 关键技术风险与对策

| 风险 | 对策 |
|------|------|
| LLM 幻觉/误判 | RAG 约束 + 知识库锚定 + 关键场景人工兜底 + "非结论"声明 + 官方核验入口 |
| 采集合规/版权 | 官方源为主、社区源只取线索并 AI 重写原创；遵守 robots 与条款；建立删除机制 |
| 2h 更新一致性/成本 | 双缓冲原子切换 + 灰度回滚 + 缓存 + 监控；待上架池预生成削峰 |
| 隐私泄露 | 字段级加密 + 自动打码 + TTL 清理 + 权限审计 |
| AI 成本 | 小模型/规则前置、结果缓存、按场景路由、私有化备选 |
| 多端排版错乱 | 服务端模板化生成 + 多分辨率回归测试（Playwright 截图比对） |
